# 日常事件追踪功能实施计划（Sidebar）

## 一、目标定义
- 在日历应用中新增「当日事件追踪」子模块，专注记录“今天”发生的所有行为事件（吃饭、上课、走路、刷手机、健身等）。
- 支持复盘与审计：每天可以看到时间花在哪里，并能回顾遗漏信息并补录。
- 功能入口放在 Sidebar，保持快速记录、低打扰、可持续使用。

## 二、需求拆解（对应 1~7）
1. 手动补录当日已完成事件（可填事件发生时间 + 持续时长）。
2. 根据当日日历事件自动入账（到点后自动生成“已发生记录”）。
3. 图表展示今日时间分配（同名事件合并统计，动态图更新）。
4. 输入事件时提供历史名称建议（自动补全/推荐）。
5. 支持“开始事件 → 稍后结束”，并允许结束时间转盘微调（开始时间精确，结束可修正）。
6. 支持“模糊记忆录入”（只记得大概时段与大概时长），系统给默认建议并可一键修正。
7. 支持删除事件。

## 三、分阶段实施计划

### Phase 1：数据模型与基础记录（先可用）
- 新增 `DailyActionEvent` 数据结构：`id`、`name`、`source`、`startAt`、`endAt`、`durationMin`、`confidence`、`createdAt`、`updatedAt`、`autoCourseKey`。
- 约束：所有事件必须属于“当前自然日”。
- 支持：新增、编辑、删除、列表展示、持久化。

### Phase 2：自动化与可视化（形成闭环）
- 自动入账：监听当日日历事件，超过结束时间后自动生成记录，避免重复入账。
- 图表：按事件名聚合 `durationMin`，输出饼图并动态刷新。
- 名称建议：基于历史频次提供候选名称。

### Phase 3：计时器与容错录入（核心体验）
- 开始/结束模式：
  - 点击“开始”即写入精确 `startAt`（立刻持久化，防止页面关闭丢失）。
  - 点击“结束”默认当前时间，可用时间转盘微调 `endAt`。
- 模糊录入模式（重点）：提供“事件名 + 大致时段 + 大致时长”的三候选快速录入。

## 四、第六点重点方案设计（模糊记忆录入）

### 方案 A：规则映射型（最快上线，稳定）
- 把“上午/中午/傍晚/吃完晚饭后/很快/一会/一下午”等描述映射到固定时间模板。
- 优点：实现快、行为可解释、调试简单。
- 缺点：个性化较弱，初期“猜你习惯”不够准。

### 方案 B：统计学习型（推荐作为主线）
- 按“事件名”累计历史统计，计算均值、中位数、P25/P75 分位值、近期窗口统计。
- 冷启动样本不足时不强制使用 B，继续用 A/AB 混合兜底。

### 方案 C：混合置信度型（长期最优）
- A + B 融合，并给每条建议打分：`score = 语义匹配分 + 历史统计分 + 最近7天习惯分`。
- 行为：
  - 高分建议直接预填。
  - 中分建议给“推荐 + 备选”。
  - 低分建议给粗粒度默认值。

## 五、建议落地顺序
1. 先做方案 A（保证马上可用）。
2. 同步埋点和数据结构，为方案 B 准备统计数据。
3. 达到数据阈值后切到方案 B 作为默认推荐逻辑。
4. 后续再迭代到方案 C（加入置信度与多候选推荐）。

## 六、默认值与词汇映射

### 时长默认桶
- 5 分钟、15 分钟、30 分钟、60 分钟、120 分钟、240 分钟

### 语义时段默认点
- 清晨 06:30
- 上午 10:00（周三/周四早八日可切换到 08:00）
- 中午 12:30
- 下午 15:00
- 傍晚 18:30
- 晚上 20:00
- 深夜 23:30

### 模糊词到时长
- 很快 = 5
- 一会 = 20
- 一阵 = 45
- 一个多小时 = 90
- 一下午 = 240

## 七、用户习惯画像（当前）
1. 早餐周中通常不吃；如果吃，多在 8 点前或 9~10 点。
2. “吃完晚饭后”通常在 19 点左右。
3. 模糊时长偏好：
   - 一会 = 10~30 分钟（中心值 20）
   - 很快 = 1~5 分钟（中心值 5）
   - 一下午 = 240 分钟
4. 周三周四有早八，上午相关候选应前移。
5. 偏好系统给 3 个候选值。
